## Summary

Chained tool calls burn context. Every tool definition sits in the prompt on every turn, and every intermediate result — the full API response, not the one field you needed — flows back through the model before the next call. A task that touches a dozen operations across a broad API pays for all of it in tokens, twice: once to load the definitions, once to relay the results. For API-heavy work, letting the agent **write code that calls the APIs** and return only the final answer inverts the economics. The model emits one script; a sandbox runs it; the model sees stdout or a return value, not a dozen round-trips.

- **The problem is round-trips, not tools**: filtering, aggregating, and looping in the model's context is what costs money — do it in code instead.
- **The pattern**: expose your tools or APIs as an importable library inside a sandbox, have the agent write a script against it, execute, and hand back only the result.
- **Convergent discovery**: Cloudflare "Code Mode", Anthropic's code execution with MCP, and Mastra's code-mode arrived at the same shape independently.
- **The catch**: generated code is untrusted and must run in a [sandbox](/docs/agents/sandboxes-and-workspaces) — this pattern does not exist safely without one.
- **The trade**: fewer tokens and cleaner loops for a weaker audit trail and harder debugging.

## The problem: chained calls relay everything through the model

A discrete tool call is a full round-trip through the context window. The model reads every tool definition to decide which to call, emits the call, waits, and then the entire tool result is inserted back into the conversation for it to reason over — even when the next step only needs one field from it. Multiply that across a task that lists records, filters them, fetches details for a subset, and aggregates: the intermediate payloads dominate the token bill, and the model's context fills with data it is merely shuttling from one call to the next.

Two costs compound here, and both are covered in more depth elsewhere. Loading hundreds of tool definitions on every turn is a [token-budget](/docs/tool-design/token-budget) and [tool-curation](/docs/tool-design/tool-curation) problem. Relaying large intermediate results is a response-shaping problem. Code execution attacks both at once by moving the orchestration — the loops, filters, and joins between calls — out of the model's context and into a script the model writes once.

## The pattern: the agent writes the orchestration

Instead of exposing each operation as a model-visible tool, expose them as functions in a code library available inside a sandbox. The agent writes a short script that imports the library, calls whatever operations it needs, does its own filtering and aggregation in ordinary code, and returns a final value. The runtime executes the script and the agent sees only what the script printed or returned.

The economic difference is the whole point. Where chained tool calls put every definition and every intermediate result in front of the model, a script puts the API surface in the sandbox and the result contract at the boundary. The model reasons about _how to solve the task in code_, not about _which of 200 tools to call next, twelve times in a row_.

### Token economics, attributed

Cloudflare reports that a representative multi-call task fell from roughly **1.17 million tokens to about 1,000 tokens** when the agent wrote a single script instead of chaining the equivalent tool calls. Treat that as a **vendor benchmark on a specific workload**, not a general result: the savings scale with how many operations the task chains and how large the intermediate results are. A task that makes two calls and returns a small payload will see little benefit; the win is concentrated in broad, data-heavy, loop-heavy work.

## When it wins

Reach for code execution when the shape of the task matches its strengths:

- **Broad APIs** — hundreds of operations where loading every definition into context is itself the cost. The sandbox holds the surface; the model imports what it needs.
- **Data-heavy intermediate results** — filter, join, and aggregate inside the script so a 10,000-row list never enters the context window; only the summary returns.
- **Repeated loops over collections** — iterating the same operation across a set is a `for` loop in code, not N identical tool-call round-trips each paying the full definition-plus-result tax.

## When it's dangerous

Generated code is untrusted code. Everything the [runtime-boundaries](/docs/runtime-boundaries) and [sandbox](/docs/agents/sandboxes-and-workspaces) guidance says about running model-authored logic applies here in full — this pattern _requires_ sandbox isolation and does not have a safe non-sandboxed form.

- **No sandbox, no pattern** — a script the model wrote can loop forever, exhaust memory, or reach the network. Run it in an isolated environment with resource limits and a network policy, never in the app process. See [Sandboxes and Workspaces](/docs/agents/sandboxes-and-workspaces) for choosing the isolation strength.
- **Secrets exposure inside the sandbox** — any credential the library needs is visible to the code and to prompt-injected instructions that code follows. Broker privileged calls through a server-side proxy the script cannot bypass, and keep production secrets out of the sandbox environment.
- **Weaker audit trail** — discrete tool calls produce a clean, per-call log; a script collapses a dozen operations into one opaque execution. Mitigate by logging the generated code verbatim and restricting the library surface to a reviewed allowlist, so what _can_ run is bounded even when what _did_ run is harder to reconstruct.
- **Debugging opacity** — a failure is now inside generated code, not at a named tool boundary. Capture stderr, the script source, and a structured error contract so a failed run is diagnosable rather than a silent empty return.

## Implementation shape

The moving parts are a library generator, a sandbox, and a result contract:

1. **Generate a typed library from your tool or OpenAPI definitions.** Each operation becomes an importable function with typed inputs and outputs. This is where MCP fits naturally: an MCP server already describes its tools with names, schemas, and descriptions, so those tools map directly to generated functions the script imports — MCP tools become importable functions rather than model-visible tools.
2. **Execute the agent's script in a sandbox.** The model returns code; the runtime runs it inside an isolated environment (see the [sandbox decision matrix](/docs/agents/sandboxes-and-workspaces)) with the generated library on the import path, resource limits set, and secrets brokered rather than injected.
3. **Return a result contract, not raw stdout.** Define what the script must return — a summary plus structured references, the same shaping rule as any [lean tool response](/docs/tool-design/token-budget) — and feed only that back into the agent loop. Capture the source and stderr alongside it for audit and debugging.

The result is that the agent loop stays small: one turn to write the script, one execution, one shaped result back — instead of a dozen definition-laden round-trips.

## Failure Modes

- **Reaching for it on narrow tasks** — a two-call task returning a small payload gains nothing and adds a sandbox, a code generator, and an opaque execution. Use discrete tool calls until breadth or intermediate-result size actually hurts.
- **Treating the sandbox as a secrets boundary** — it contains what the code can _damage_, not what it can _read_. A credential in the environment is readable by any script and any injection that steers it. Broker privileged actions server-side.
- **Unbounded library surface** — exposing your entire internal API to generated code turns one bad script into a wide blast radius. Restrict the importable surface to a reviewed allowlist scoped to the task.
- **Discarding the generated code** — if you log only the result, a wrong answer is unreconstructable. Log the script source with the execution so the audit trail survives the collapse from N calls to one.
- **Swallowed failures** — a script that catches its own errors and returns an empty result hides breakage from the loop. Enforce a structured error contract (stderr + reason) so a failed execution surfaces as a failure, not a plausible-looking blank.
