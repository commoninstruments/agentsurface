## Summary

An agent loop is non-deterministic: the same input produces different model output, tool results vary, and the number of steps is not known in advance. A durable workflow engine assumes the opposite — that replaying recorded history reproduces the same code path exactly. Wrap an agent loop naively in a workflow and the first replay after a crash diverges from history and the run fails.

Durable execution reconciles the two. The model calls and tool invocations run as recorded steps whose results are memoized, so replay reads decisions from history instead of re-rolling them. Everything the engine replays stays deterministic; everything non-deterministic hides behind a recorded boundary.

- **Non-determinism lives in steps** — every model call and tool call is a recorded, memoized unit.
- **Partial agent state is checkpointed** — conversation, tool results, and loop position survive a restart.
- **Long pauses are first-class** — suspend for hours-long human approval without holding a process open.
- **Runs can be steered mid-flight** — new instructions inject into a running loop through workflow events.
- **History is the debugger** — replay reconstructs exactly what the agent saw and did.

## When An Agent Needs Durable Execution

Reach for a durable workflow engine, not a plain background job, when the agent run:

- outlives a single request or function invocation (minutes to days)
- must survive process restarts, deploys, and disconnects without losing progress
- pauses for human approval measured in hours, not seconds
- fans work out to sub-agents or tools that each retry independently
- needs an exact, replayable record of every decision for audit or debugging

A plain background job (a cron worker, a queue consumer) is enough when the run is short, idempotent, and safe to restart from the beginning. Scheduling, batching, and idempotency-key mechanics for that simpler case live in [Autonomous Background Agents](/docs/cookbook/autonomous-background-agents); this page is about the harder case where restarting from the beginning is not acceptable.

Decision rule: if losing an in-flight run means re-paying for completed model calls or re-running irreversible side effects, the loop belongs in durable execution.

## Implementation Shape

The mechanics are the same across engines; only the primitives are named differently.

**Record every non-deterministic call as a step.** The loop's control flow — branch on the result, decide whether to continue — is deterministic and replays for free. The model call and each tool call are wrapped so their results are written to history and memoized on replay.

```ts
// Vendor-neutral shape: the loop is deterministic; the calls are recorded.
async function agentRun(ctx, input) {
  let state = ctx.restoreOrInit(input); // checkpointed partial state
  while (!state.done && state.steps < MAX_STEPS) {
    // Recorded: replays from history, never re-rolls the model.
    const decision = await ctx.step("model", () => callModel(state.messages));
    if (decision.toolCall) {
      const result = await ctx.step(`tool:${decision.toolCall.name}`, () =>
        runTool(decision.toolCall),
      );
      state = ctx.checkpoint(applyToolResult(state, result));
    } else {
      state = ctx.checkpoint(applyFinal(state, decision));
    }
  }
  return state.output;
}
```

**Checkpoint partial state.** Persist the conversation, accumulated tool results, and loop position after each step so a crash resumes mid-run instead of restarting the whole loop.

**Suspend and resume for approvals.** When a step needs a human decision, the workflow suspends and releases its worker; an external event — approval, rejection, edit — resumes it hours later. Who can approve, what they see, and how a rejection rewinds are approval semantics covered in [Human-in-the-Loop](/docs/multi-agent/human-in-the-loop).

**Signal liveness for long calls.** A long-running tool call — a multi-minute browser session, a slow model — must periodically signal progress so the engine does not declare it dead and retry it. This is a vendor-neutral concern; Temporal implements it as activity heartbeats.

**Inject signals mid-run.** Steering a running agent — new instructions, a cancel, a priority change — arrives as an event the loop drains between steps. This is the workflow-primitive side of steering; the UX side belongs to session control. Temporal calls these Signals, Inngest and Cloudflare model it as event-driven resume, and Mastra exposes agent signals (beta).

**Replay as time-travel debugging.** Because history records every input the agent saw, a failed run can be replayed step by step against current code — the payoff that justifies the constraints above.

## Engine Anchors

Each engine solves the same problem with its own primitives. Treat these as equal-weight options, not a ranking.

**Temporal** — activities are the recorded, memoized unit; Signals inject mid-run input; Queries read live state; replay is the core testing model, and heartbeats keep long activities alive. The June 2026 Standalone Activity primitive records a single durable step without authoring a full workflow — convenient for wrapping one expensive model call.

```ts
// Temporal: the agent step is an activity; the loop is workflow code.
const { callModel, runTool } = proxyActivities({
  startToCloseTimeout: "5m",
  heartbeatTimeout: "30s",
});
setHandler(steerSignal, (msg) => pendingSteer.push(msg)); // mid-run injection
```

**Inngest** — `step.run()` memoizes each step's result; a replay skips completed steps and resumes at the first incomplete one. `step.waitForEvent()` suspends for approvals or steering. This is the same memoization that powers Mastra's durable agents.

```ts
inngest.createFunction({ id: "agent-run" }, { event: "agent/start" }, async ({ step }) => {
  const decision = await step.run("model", () => callModel(messages));
  const approval = await step.waitForEvent("approval", {
    event: "agent/approved",
    timeout: "24h",
  });
});
```

**Cloudflare Workflows** — steps are durable and retried independently; a workflow instance can sleep or wait for an event across restarts, running close to the model at the edge.

**AWS (LangGraph + DynamoDB)** — the framework-native pattern: LangGraph drives the agent graph, a DynamoDB checkpointer persists graph state after each node, and the run resumes from the last checkpoint. Durability comes from the checkpoint store, not a dedicated workflow engine.

**Mastra durable agents** — one framework option that layers durable execution over the agent loop, built on Inngest-style step memoization (beta). A convenient default when already in Mastra, not the only shape.

## Failure Modes

- **Non-replay-safe side effects in workflow code.** A wall-clock read, a random value, a direct network call, or a UUID generated in the deterministic loop body diverges on replay. Every such call must move into a recorded step.
- **Unbounded history growth.** A chatty agent loop with hundreds of steps bloats workflow history until replay is slow or the engine rejects it. Cap steps, summarize and restart the run (continue-as-new), or offload large payloads to storage and record references instead.
- **Checkpoint staleness against long tool calls.** A tool that runs for minutes while the checkpoint still reflects pre-call state means a crash mid-tool loses that progress. Checkpoint before starting irreversible work and make the tool itself idempotent.
- **Signal races.** A steer signal that arrives between a model call and its tool execution can be read too late or applied to a stale decision. Define exactly where in the loop signals are drained, and make injected instructions idempotent against double-delivery.
