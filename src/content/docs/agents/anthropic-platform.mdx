## Summary

Anthropic's agent surface is broader than "Claude plus tool use." By July 2026, treat Claude as a platform: Anthropic-run Managed Agents, the Claude Code harness across four surfaces, the Claude Agent SDK for embedding that harness, cross-vendor Agent Skills, an MCP connector in the Messages API, hosted built-in tools, and a fast-moving model and runtime layer.

The common failure is scaffolding against last quarter's shape — a renamed SDK, a superseded model ID, or a deprecated request parameter. Use this page when evaluating or scaffolding Claude-native systems, and check the freshness stamp above before trusting any dated claim.

Anthropic docs now live at `platform.claude.com` (`docs.anthropic.com` redirects there); Claude Code docs live at `code.claude.com`.

## Managed Agents

Managed Agents has been in public beta since 2026-04-08. Anthropic runs the loop; you model the work with four concepts:

- **Agents** — the configured harness (instructions, tools, model).
- **Environments** — where execution happens: Anthropic's cloud, or a self-hosted sandbox you control (added 2026-05-19).
- **Sessions** — a single run with lifecycle events, memory, vault-backed credentials, and webhooks.
- **SSE events** — server-sent event streams for progress, tool calls, and outcomes.

Later beta additions include cron-scheduled deployments (2026-06-09) for recurring agents and MCP tunnels for reaching private MCP servers from a managed run. Public pricing for Managed Agents is not something to quote here — session-hour figures reported in press coverage are unverified.

## Decision Guide

Choose **Claude Managed Agents** when:

- sessions, lifecycle events, memory, or outcomes should be managed by Anthropic
- the agent needs a managed cloud environment or a self-hosted sandbox rather than an in-app loop
- credential vaulting, webhooks, or scheduled (cron) runs matter
- you want a cloud agent service rather than embedding the loop in your own app server

Choose **Claude Code / the Claude Agent SDK** when:

- the job is coding, repository inspection, editing, test execution, or incident response
- a human works interactively across terminal, IDE, desktop, or browser (Claude Code)
- a product needs to invoke that harness programmatically with permission modes, allowed/disallowed tools, and local/project/user MCP configuration (Agent SDK)
- the pattern is "prototype in the Agent SDK, then promote to Managed Agents for production operation"

Choose **Messages API plus MCP connector** when:

- the request only needs tool calls to one or more public remote MCP servers
- the agent loop can stay inside a single Messages API interaction
- you do not need managed sessions, memory, environments, outcomes, or webhooks

Choose **Agent Skills** when:

- the system needs procedural know-how, templates, scripts, examples, or domain assets
- the work is repeatable and larger than a single tool call
- a Skill can teach the agent how to use existing tools rather than adding another API surface

Choose **custom MCP servers** when:

- Claude, OpenAI, Gemini, Cursor, Codex, and other clients should all use the same external capability
- the system needs a vendor-neutral tool/resource/prompt contract
- auth, scopes, audit, pagination, and structured results must be server-owned

## MCP Notes for Claude

Claude has multiple MCP paths and they are not identical:

- Claude Code can use local stdio and remote HTTP/SSE MCP servers through `.mcp.json` and CLI configuration.
- Claude Code can also expose its own tools as an MCP server with `claude mcp serve`; the client connecting to it must implement user confirmation for risky calls.
- The Claude Agent SDK supports MCP configuration and allowed-tool filtering.
- The Messages API MCP connector uses the `mcp-client-2025-11-20` revision header, supports `mcp_toolset` with `defer_loading` to keep large tool catalogs out of the initial context, and accepts OAuth bearer tokens — but its documented limitation is that only MCP tool calls are currently supported.

When documenting or scaffolding MCP for Claude, always state which path is being used. Do not assume support for resources, prompts, roots, sampling, elicitation, or tasks unless the selected Claude surface documents it.

## Agent Skills

Skills package repeatable workflows as folders of instructions, scripts, and resources. They are now a **cross-vendor open standard** published at `agentskills.io`, with 40+ client adopters rather than a Claude-only mechanism; `skills.sh` is the de facto marketplace. (There is no `skills.claude.com`.)

Treat Skills as complementary to MCP: MCP exposes callable systems, Skills teach an agent how to carry out repeatable work with the tools it already has. Because Skill content is model-visible, keep credentials and authorization out of it.

## Tooling Guidance

Prefer Claude built-in tools when they match the job:

- web search or web fetch for current public web context
- code execution for data processing, generated-code checks, and tool-efficient workflows
- tool search for large tool catalogs
- memory tool or Managed Agents sessions for cross-session recall
- Agent Skills for packaged procedures

Prefer custom tools or MCP when the capability is product-specific, authenticated, auditable, cross-vendor, or needs strict schemas and server-side authorization.

## Model Guidance

Use current model discovery and migration docs rather than hard-coding old IDs. As of 2026-07-06:

- **Fable 5** (`claude-fable-5`) is the flagship, GA since 2026-06-09.
- **Opus 4.8** (`claude-opus-4-8`) is the recommended default for most agent and coding work.
- **Sonnet 5** (`claude-sonnet-5`) released 2026-06-30 as the mid-tier workhorse.
- **Haiku 4.5** remains the low-latency, low-cost option.
- **Sonnet 4 and Opus 4 were retired on 2026-06-15** — that retirement is executed, not pending. Do not scaffold against those IDs.
- **Opus 4.1 retires on 2026-08-05.** Migrate off it before then.

Runtime changes that affect cost, latency, and request shape:

- From the 4.6 generation onward, pinned model IDs are **dateless** (for example `claude-opus-4-8`). Do not append a date suffix to current IDs.
- A **new tokenizer** ships on Opus 4.7 and later, Fable 5, and Sonnet 5. The same text produces roughly 30% more tokens, so recompute cost and context budgets instead of reusing older estimates.
- On **Opus 4.7 and later**, `temperature`, `top_p`, and `top_k` are deprecated and return a 400 error. Use the effort and thinking controls to tune cost and latency instead.
- Prefer model discovery over hard-coded assumptions where the app can query available models and their capabilities.

## Security Guidance

Claude-native systems still need server-side policy:

- Treat model-visible tool descriptions, MCP annotations, resources, and Skill content as advisory inputs.
- Enforce authorization, tenant identity, and scope checks in trusted server code.
- Gate destructive, external, authenticated, production, browser, and sandbox tools with confirmation policies.
- Store credentials in vaults or environment-backed secret managers, not prompts or Skill text.
- Log agent ID, session ID, user/tenant context, tool calls, approvals, and outcomes.
