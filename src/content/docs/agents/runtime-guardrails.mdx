## Summary

Red-teaming finds holes before launch. Guardrails are what stands in the request path in production, after the attacker you never modeled shows up. An agent system without runtime guardrails is relying on the model to refuse — and a model refusal is a behavior, not a control. It varies by prompt, drifts across model versions, and cannot be audited.

Runtime guardrails are deterministic processors that wrap the agent loop: they inspect input before the model sees it and inspect output before it reaches the user or a tool. They fail loudly, log every decision, and do not depend on the model choosing to cooperate.

- **Guardrails are controls, model refusals are not**: put enforcement in code that runs every time.
- **Two-sided pipeline**: input processors sanitize what goes in; output processors gate what comes out.
- **Ordered and budgeted**: each check costs latency and money — sequence them and cap them.
- **Fail-open or fail-closed is a per-processor decision**: a moderation timeout and a PII-scrub timeout are not the same risk.
- **Not a substitute for authorization**: guardrails filter content; they do not decide who may do what.

## The Problem

The offense side of this coin is [red-teaming](/docs/testing/red-teaming): enumerating prompt injection, data exfiltration, tool abuse, and jailbreaks, then proving your system resists them in CI. That work tells you _what can go wrong_. It does not stand in the production request path.

Runtime guardrails are the defense side. When a real request arrives — from a user, a retrieved document, or another agent's tool output — something has to inspect it before the model acts, and inspect the model's response before it leaves the system. If that "something" is the model's own judgment, you have no control: it cannot be tested in isolation, it changes when you upgrade the model, and it produces no audit trail.

A guardrail is a processor that runs unconditionally around the loop. It is code you can unit-test, latency you can measure, and a decision you can log.

## Decision Rule

Which guardrails you need is set by two variables: the **deployment shape** of the agent and the **sensitivity of the data** it touches.

| Deployment shape                                        | Primary risk                                        | Minimum guardrails                                                                                     |
| ------------------------------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| User-facing chat (public or semi-trusted input)         | Prompt injection, abusive content, brand-safety     | Input: injection detection, content moderation. Output: response moderation, system-prompt-leak scrub. |
| Internal tool-calling agent (trusted users, real tools) | Data exfiltration, over-broad tool use, cost blowup | Input: PII redaction on logged input. Output: cost guard, PII scrub, around-tool argument checks.      |
| Autonomous background agent (no human in the loop)      | Runaway cost, unbounded loops, silent data leaks    | Output: hard token/cost limiters, per-run budgets. Input: injection detection on all retrieved data.   |

Then layer data sensitivity on top:

- **Regulated or personal data in scope** (PII, PHI, payment data): PII detection and redaction becomes mandatory on both sides, and it usually runs **fail-closed** — if the detector is unavailable, block rather than leak.
- **Public, low-stakes content**: moderation can run **fail-open** — a moderation-service timeout should not take down the whole product.

The practical rule: pick the guardrails the deployment shape demands, then upgrade each one to fail-closed wherever the data it protects is sensitive.

## Implementation Shape

Model the guardrails as an ordered pipeline with the agent loop in the middle. Input processors run left-to-right before the model; output processors run before the response is returned or a tool is invoked.

```
request
  → [input] prompt-injection detection
  → [input] PII detection / redaction
  → [input] content moderation
  → [input] language detection
      → AGENT LOOP (model + tools)
  → [output] cost guard / token limiter
  → [output] response moderation
  → [output] PII scrubbing
  → [output] system-prompt-leak scrubbing
response
```

### Input processors

- **Prompt-injection detection** — classify user input and retrieved tool output for embedded instructions before the model reads them. Treat retrieved documents and prior tool results as untrusted, not just the top-level user message.
- **PII detection and redaction** — strip or tokenize personal data _before_ the model sees it, so sensitive values never enter prompts, logs, or provider request bodies.
- **Content moderation** — reject abusive or disallowed input early, before spending a model call.
- **Language detection** — route or reject based on supported languages; unexpected languages are also a common injection vector.

### Output processors

- **Cost guards and token limiters** — cap tokens per request and spend per run. For autonomous agents this is the load-bearing control, not a nicety.
- **Response moderation** — check generated text against your content policy before it reaches the user.
- **PII scrubbing** — catch personal data the model may have reconstructed or echoed.
- **System-prompt-leak scrubbing** — detect and strip fragments of the system prompt or internal instructions from responses.

### Where each check runs

Three placement points matter, and they are not interchangeable:

1. **Before the model** — input processors. Cheapest place to reject; nothing downstream runs.
2. **After the model** — output processors on the final response.
3. **Around tool calls** — validate tool _arguments_ before execution and tool _results_ before they re-enter the context. This is where injection-via-tool-output and exfiltration-via-tool-call are actually stopped, and it overlaps with [tool safety and idempotency](/docs/tool-design/idempotency-and-safety).

### Latency and cost budget

Every processor adds latency, and model-based detectors (an injection classifier is often itself a model call) add real cost. Give the pipeline a budget:

- Order processors cheapest-and-most-selective first, so most rejections happen before the expensive checks.
- Run independent checks concurrently where the framework allows, rather than strictly serially.
- Prefer a fast local classifier or regex/entropy pass ahead of a network call to a hosted service.

### Fail-open versus fail-closed

Decide this per processor, and write it down.

<Callout type="warn">
  A guardrail's failure mode is a security decision, not a default. Fail-closed (block on error) for
  PII redaction and injection detection on sensitive data — an unavailable detector must not become
  an open door. Fail-open (allow on error) is defensible for best-effort moderation on low-stakes
  content, where a hard dependency on a third-party service would make it a single point of failure
  for the whole product.
</Callout>

### Vendor landscape

Several systems provide guardrail primitives you compose rather than build from scratch. Treat them as roughly equivalent starting points and choose by runtime and data-residency needs, not feature-list length:

- **NVIDIA NeMo Guardrails** — programmable rails (Colang) for input, output, and dialog flow.
- **Guardrails AI** — a validator framework with a hub of composable input/output validators.
- **AWS Bedrock Guardrails** — managed policies (content filters, denied topics, PII redaction, contextual grounding) applied around Bedrock calls.
- **Azure AI Content Safety** — hosted moderation and prompt-shield detection.
- **Lakera Guard** — a hosted API focused on prompt-injection and data-leak detection.

Frameworks also ship built-in pipelines: [Mastra](/docs/agents/framework-selection) exposes processors (pii-detector, prompt-injection-detector, moderation, cost-guard, token-limiter) as ordered input/output steps around the agent, which is one concrete shape of the pipeline described above.

## Failure Modes

- **Latency stacking** — each processor is individually fast; six of them in series are not. Budget the pipeline, run independent checks concurrently, and put the cheap rejectors first.
- **False-positive lockouts** — an over-tuned injection or PII detector blocks legitimate work (a security engineer discussing `rm -rf`, a doctor entering real patient context). Measure false-positive rate on real traffic, not just recall on attacks.
- **Pattern-only injection detectors** — detectors that match known jailbreak strings catch yesterday's attacks and miss novel phrasings. Combine classifiers with structural defenses (untrusted-data separation, tool-argument validation) rather than trusting a denylist.
- **Moderation mistaken for authorization** — a content filter decides _whether text is allowed_, never _whether this principal may perform this action_. Authorization belongs in [tool design and identity](/docs/tool-design/idempotency-and-safety), enforced server-side. A clean moderation pass is not permission.
- **Cost guards that kill legitimate long runs** — a flat token cap that stops a valid multi-hour background job is an outage, not a control. Budget per run and per task class, and surface a structured "budget exceeded" error the caller can resume from, rather than a silent kill.
- **Unlogged decisions** — a guardrail that blocks without emitting a trace ID and reason is unauditable and undebuggable. Every allow/block decision should be logged.
