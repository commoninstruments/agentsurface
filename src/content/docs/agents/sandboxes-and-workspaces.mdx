An agent that writes and runs code, manipulates files, or executes model-generated logic needs isolation the host process cannot safely provide. Generated code can loop forever, exhaust memory, read environment secrets, write outside its intended directory, or reach the network. Running it inside the app server puts all of that on the same process that holds your database credentials and serves user traffic.

A sandbox moves that execution into a disposable environment with its own filesystem, process table, and network policy, so a bad run is contained and torn down instead of taking the app with it. A workspace is the next question: long-lived agents also need state that survives across sessions — a filesystem, a code index, searchable memory, and a skills library that persist between runs rather than being rebuilt each turn.

This page is the decision-guidance counterpart to [Browser Access](/docs/agents/browser-access). The full vendor comparison lives in the [tooling catalog](/docs/tooling-catalog/browser-sandbox-and-integration); this page decides _whether_ and _what kind_.

## Decision Matrix

| Need                                                          | Prefer                                          | Why                                                                                                     |
| ------------------------------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------------------------------------------------- |
| Agent calls known APIs and tools, never runs code             | No sandbox                                      | If nothing untrusted executes, isolation is overhead. Keep tools server-side and deterministic.         |
| Occasional evaluation of model-generated code or data scripts | Ephemeral sandbox                               | Spin up, run, capture output, tear down. No persistence means no state to leak between calls.           |
| A coding-agent session that edits and runs a repo             | Sandbox plus a persistent filesystem            | The agent needs the same working tree across many turns; a fresh sandbox each turn loses its edits.     |
| A long-lived autonomous agent that learns over time           | A full workspace (filesystem, search, skills)   | Durable memory, a code index, and a skills library outlive any single session and compound across runs. |
| Untrusted or adversarial code, multi-tenant isolation         | microVM-backed sandbox                          | A hardware-virtualized boundary is the appropriate trust level when the code itself may be hostile.     |
| Trusted first-party code, latency-sensitive short tasks       | Container or process sandbox close to the agent | Weaker isolation, but far lower cold-start; acceptable when you control the code that runs.             |

Two axes drive the choice. The first is **what persists**: nothing (ephemeral), the filesystem (coding sessions), or the whole workspace (autonomous agents). The second is **isolation strength versus cold-start latency**:

- **microVM** (Firecracker-style hardware virtualization, as used by Vercel Sandbox and E2B) — strongest isolation, suitable for untrusted code, at the cost of a heavier cold start.
- **container** (Cloudflare Sandbox, AgentCore) — good isolation with a lighter footprint, well-suited to first-party generated code.
- **process** (in-VM subprocess, seccomp/namespace jails) — lowest latency, weakest boundary; only for code you trust and can bound.

Choose the weakest isolation that still contains the worst case for the code you actually run. Do not pay microVM cold-start for a trusted 200ms transform, and do not run genuinely untrusted code in a shared process.

## Provider Landscape

Treat these as roughly interchangeable starting points and choose by where your agent already runs, not by feature-list length. The [tooling catalog](/docs/tooling-catalog/browser-sandbox-and-integration) holds the full comparison table; the characterizations below are decision-level only.

- **E2B** — secure sandboxes and code-interpreter infrastructure positioned around enterprise agent-cloud deployments, including secure MCP execution inside the sandbox. Strong fit for data analysis and generated-code execution.
- **Daytona** — hosted sandbox product remains active; the open-source repository was frozen in June 2026, so treat the OSS path as unmaintained and prefer the hosted offering.
- **Modal** — serverless compute with VM-backed sandboxes (June 2026), useful for elastic Python and GPU workloads; integrates with Anthropic's Claude Managed Agents.
- **Vercel Sandbox** — ephemeral microVMs for running generated or untrusted code, GA on 2026-01-30, with filesystem persistence reaching GA on 2026-05-26. Fits agents already on Vercel infrastructure.
- **Cloudflare Sandbox SDK and Containers** — container sandboxes for command execution, files, terminals, and code-interpreter workflows driven from a Worker, GA on 2026-04-13. Strong fit for edge-hosted agents.
- **AWS Bedrock AgentCore** — managed Runtime and Code Interpreter for AgentCore agents; the right choice when the agent stack is intentionally AWS-native.
- **Anthropic Managed Agents** — self-hosted sandboxes are part of the Managed Agents platform, keeping execution inside your own boundary while Anthropic manages the agent runtime.

## Workspaces Beyond The Sandbox

An ephemeral sandbox is enough for a one-shot execution. A long-running agent needs a workspace: durable state that outlives any single session and makes the agent more capable over time. Frame this vendor-neutrally — the capabilities matter more than the product.

- **Persistent agent filesystems** — provider-backed storage (S3/GCS-style object mounts, or layered agent-fs snapshots) that the sandbox mounts on each session, so a coding agent resumes on the same working tree instead of re-cloning. Persistence needs a lifecycle: what is kept, for how long, and who can read it.
- **LSP-backed code inspection** — a language server in the workspace lets the agent navigate code semantically — definitions, references, types, call hierarchies — instead of falling back to `grep`. Semantic navigation both reduces token spend and produces more precise edits than text search.
- **Workspace search** — BM25, vector, or hybrid retrieval over the workspace's own contents (files, notes, prior results), so the agent recalls what it learned rather than re-deriving it. Hybrid keyword-plus-vector retrieval is usually the pragmatic default; see [Data Retrievability](/docs/data-retrievability) for the retrieval mechanics.
- **Skills libraries in the workspace** — reusable procedures the agent has authored or been given, stored in the workspace and loaded on demand, so learned capability compounds across sessions.

[Mastra](/docs/agents/framework-selection) Workspaces is one implementation that bundles a persistent filesystem, code inspection, and workspace search behind a single interface; the pattern is not specific to it.

## Failure Modes

- **State leaking between sessions expected to be isolated** — reusing a sandbox or filesystem across tenants or tasks that assumed a clean environment leaks data and corrupts results. Isolate by tenant and task class, and prove teardown actually happens.
- **Cold-start dominating short tasks** — a microVM that takes seconds to boot in front of a sub-second transform makes the isolation the slowest part of the run. Match isolation strength to the threat, pool warm sandboxes, or run trusted short work in a lighter boundary.
- **Persistence without a retention policy** — a workspace filesystem that only ever grows becomes an unbounded, unaudited store of scraped data, secrets, and intermediate artifacts. Set retention, size limits, and access scope before turning persistence on.
- **Treating the sandbox as a secrets boundary** — a sandbox contains what the code can _damage_, not what it can _read_. Any secret injected into the environment is visible to the code and to prompt-injected instructions that code follows. Keep production credentials out of the sandbox and broker privileged actions through a server-side tool the sandbox cannot bypass.
