## Summary

Every generated tool has three fields ŌĆö `name`, `description`, and input `schema` ŌĆö and each maps mechanically from one OpenAPI operation. This page covers only that mapping: how an operation's `operationId`, `summary`, parameters, request body, and responses become a tool definition. The portable rules for _writing_ good names, descriptions, schemas, and curated tool sets are not API-specific and live in the [Tool Design](/docs/tool-design) section ŌĆö this page links there rather than repeating them.

- `operationId` ŌåÆ tool `name`
- `summary` / `description` ŌåÆ tool `description`
- `parameters` + `requestBody` schema ŌåÆ input `schema`
- `2xx` response schema ŌåÆ output schema
- `x-agent-*` extensions ŌåÆ MCP annotations

## Operation-to-Tool Mapping

When a framework ingests your OpenAPI spec, it generates one tool per operation using this mapping:

| OpenAPI element           | Tool field    | Notes                                                                  |
| ------------------------- | ------------- | ---------------------------------------------------------------------- |
| `operationId`             | `name`        | Must be globally unique and `verb_noun` ŌĆö it becomes the function name |
| `summary` + `description` | `description` | The agent's routing prompt; write it as trigger conditions, not docs   |
| path/query `parameters`   | input schema  | Flattened into the schema's `properties`                               |
| `requestBody` schema      | input schema  | Merged with parameters into one flat object                            |
| `2xx` response schema     | output schema | Enables downstream tools to reason over typed results                  |
| `x-agent-*` extensions    | annotations   | Confirmation, idempotency, destructive hints                           |

Because the generated `name` and `description` come straight from `operationId` and `summary`, spec quality is tool quality. A vague `summary` becomes a vague routing prompt; a missing `operationId` breaks tool generation entirely. See [OpenAPI for Agents](./openapi-for-agents) for the spec-authoring rules that make this mapping produce good tools.

## Worked Example

This operation:

```yaml
paths:
  /orders/{orderId}:
    get:
      operationId: get_order
      summary: >
        Retrieves full details for a single order including line items and
        shipping status. Use when you have an order ID. For searching orders,
        use search_orders instead.
      parameters:
        - name: orderId
          in: path
          required: true
          schema:
            type: string
          description: The order identifier returned by list_orders or create_order
      responses:
        "200":
          description: The order
          content:
            application/json:
              schema:
                $ref: "#/components/schemas/Order"
```

Generates this tool definition:

```json
{
  "name": "get_order",
  "description": "Retrieves full details for a single order including line items and shipping status. Use when you have an order ID. For searching orders, use search_orders instead.",
  "inputSchema": {
    "type": "object",
    "required": ["orderId"],
    "properties": {
      "orderId": {
        "type": "string",
        "description": "The order identifier returned by list_orders or create_order"
      }
    }
  }
}
```

## Extensions and Annotations

Agent-facing behavior that has no native OpenAPI field is carried by `x-agent-*` extensions on the operation and translated into MCP annotations (`readOnlyHint`, `idempotentHint`, `destructiveHint`) at generation time. The extension-to-annotation table lives in [OpenAPI Extensions](./openapi-extensions); the semantics of the annotations themselves are covered in [Tool Design: Idempotency and Safety](/docs/tool-design/idempotency-and-safety).

## Checklist

Before generating tools from your API:

- [ ] Every operation you want exposed has a unique `verb_noun` `operationId`
- [ ] Each `summary` reads as an agent trigger condition, not human documentation
- [ ] Path and query parameters carry `description` and constraints
- [ ] Request bodies reference reusable component schemas
- [ ] `2xx` responses declare a schema so output types survive generation
- [ ] Destructive or non-idempotent operations carry the matching `x-agent-*` extension
- [ ] You expose a curated subset, not one tool per endpoint (see [Tool Curation](/docs/tool-design/tool-curation))
